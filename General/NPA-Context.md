# Netskope Private Access (NPA) — General Context

## Overview

| Component | Description |
|---|---|
| **NPA Publisher** | Lightweight on-prem connector that establishes a secure outbound tunnel to the Netskope cloud |
| **Traffic flow** | `Netskope Cloud → NPA Publisher → Internal Application` |
| **Runtime** | C++ business logic running inside a container on a Linux Ubuntu host |
| **Category** | Zero Trust Network Access (ZTNA) — similar products: Zscaler Private Access (ZPA), Cloudflare Access |

The NPA Publisher is deployed inside the private network, exposed only to the Netskope Cloud. Internal applications are reachable exclusively through the Publisher.

---

## Data Path — Traffic Flow

> Scope: **data path only** (not enrollment, not Routing Policy generation APIs).

### 1. Forward Path — Client → Private App Server

Once the client is enrolled, has a Routing Policy, and has a tunnel to a Gateway:

**Step 1 — App traffic classified on the client**

- User opens a private app (e.g. `app.corp.internal:443` / SSH / RDP).
- Client checks: does this flow match a **private app entry** in the steering config / Routing Policy?
  - **YES** → packet is **not** sent directly to destination; it is encapsulated and sent over the **secure tunnel to the Cloud Gateway**.

**Step 2 — Gateway enforces Routing Policy**

> The management plane has already generated the Routing Policy based on customer policy and app definitions.

- Gateway looks up the traffic in the **Routing Policy** generated by the Policy Orchestrator:

  | Routing Policy Action | Gateway Behavior |
  |---|---|
  | **Allow / Forward** | Forwards into the data-plane path toward Session Relay / Publisher |
  | **Deny** (implicit or explicit, no template) | Drops the traffic |
  | **Block with template** | Drops traffic + triggers block/notification behavior via `block_notification_config`; nothing reaches publishers |

**Step 3 — Gateway → Session Relay**

- Gateway picks the appropriate **Session Relay** in the correct DP region.
- Traffic travels over the **Gateway–Session Relay** channel inside the Cloud Data Plane.

**Step 4 — Session Relay → Publisher**

- Session Relay resolves the **service → publisher mapping** from the Policy Orchestrator (`service_publisher_assignments`):
  - **Primary publisher** — used when healthy.
  - **Secondary publisher** — failover when primary is unavailable.
- Traffic is forwarded to the Publisher over its established control/data channel.

**Step 5 — Publisher → Private App Server**

- Publisher is deployed close to or inside the customer network (VPC, DC, etc.).
- It has a regular L3/L4 route to the private app server:
  - Resolves the app host/IP.
  - Opens/maintains TCP/UDP sessions to the app on the configured port(s).
- From the app server's perspective, this is a standard inbound connection from a local IP (the Publisher).

**Forward path summary:**

```
Client → Cloud Gateway → Session Relay → Publisher → Private App Server
```

---

### 2. Reverse Path — Private App Server → Client

The reverse direction follows the **same chain in reverse**, bound to the same session:

**Step 1 — Private App Server → Publisher**

- App server sends responses to the IP/port of the Publisher.
- From the server's POV, Publisher is just a normal client on the same network.

**Step 2 — Publisher → Session Relay**

- Publisher sends packets back into the existing **Publisher–Session Relay** tunnel.
- Packets are tagged with the correct session context (service + client session).

**Step 3 — Session Relay → Gateway**

- Session Relay forwards to the same **Gateway** that owns that client session.
- No new policy evaluation — this is the reverse of an already-allowed flow.

**Step 4 — Gateway → Client**

- Gateway sends packets down the **client tunnel**.
- Client decapsulates and passes them up the OS/TCP stack to the user application.

**Reverse path summary:**

```
Private App Server → Publisher → Session Relay → Gateway → Client
```

---

### 3. Where "Routing" vs "Policy" Live

#### Control Plane — Policy & App Visibility

| Component | Responsibility |
|---|---|
| **Policy Engine** | Decides *which private apps* a user/device can access |
| **Routing Policy config API** | Encodes Allow / Block with template / implicit Deny per app |

#### Data Plane — Routing & Packet Forwarding

| Component | Responsibility |
|---|---|
| **Policy Orchestrator** | Builds the Routing Policy: app definitions (host/port/protocol), Forward/Deny/Block actions, publisher assignments (primary/secondary) |
| **Gateway** | Applies Routing Policy; sends allowed flows to Session Relay |
| **Session Relay** | Picks the correct Publisher for each app/session |
| **Publisher** | Uses normal internal routing to reach the app server |

**End-to-end chain:**

```
Client ↔ Cloud Gateway ↔ Session Relay ↔ Publisher ↔ Private App
```

# Let’s go step by step.

---

## 1. Concrete example: SSH to an IP‑based private app

Assume:

- Private app: `10.10.20.15:22` (an internal Linux server)
- User: on a laptop with NPA client installed
- NPA is already fully set up (publishers running inside the customer network, Routing Policy generated, etc.)

I’ll mark *who* is acting at each step: **Client**, **Gateway**, **Session Relay**, **Publisher**, **App Server**.

### 1.1. Control‑plane setup (done once per user/device/session)

**(a) Client connects to NPA**

1. **Client → Gateway**
   - Client establishes a secure tunnel (TLS/IPsec/whatever is used) to **Cloud Gateway**.
   - It authenticates using the **certificate** it got earlier from Cloud Admin.
2. **Gateway → Policy Orchestrator → Policy Engine → Policy Orchestrator**
   - Gateway asks Policy Orchestrator: “Give me the Routing Policy for this user/device.”
   - Policy Orchestrator calls the **Policy Engine** to get:
     - Which private apps this user/device is allowed to access
     - For each app: allow/block/deny, publisher assignments, notification template, etc.
   - Policy Orchestrator builds a **Routing Policy**:
     - Contains one entry for our SSH app, something like:
       - `dst_ip = 10.10.20.15`  
       - `dst_port = 22`  
       - `protocol = TCP`  
       - `action = FORWARD`  
       - `publishers = [Publisher-A (primary), Publisher-B (secondary)]`
   - Policy Orchestrator sends Routing Policy back to Gateway.
3. **Gateway → Client**
   - Gateway pushes Routing Policy to the client over the management/control channel.
4. **Client**
   - Stores Routing Policy/steering info locally:
     - “If user opens TCP:22 to 10.10.20.15, send into NPA tunnel, not directly.”

Now the system is primed to route SSH to `10.10.20.15` through NPA.

---

### 1.2. Data‑plane: client → app server (SSH connect)

#### Step 1: User starts SSH

1. **User → Client**
   - User runs: `ssh admin@10.10.20.15`
   - OS tries to open a TCP connection to `10.10.20.15:22`.

2. **Client flow classification (on device)**
   - NPA client intercepts the outbound TCP SYN.
   - It checks against Routing Policy/steering:
     - Destination IP: `10.10.20.15`
     - Destination port: `22`
     - Protocol: TCP
   - Finds a **matching Routing Policy rule** with action = FORWARD.
   - Instead of sending SYN directly to `10.10.20.15`, it:
     - Encapsulates it
     - Sends it through the tunnel to the **Gateway**.

So packet flow so far:  
**User SSH → Client (intercept) → Gateway (via secure tunnel)**

#### Step 2: Gateway enforces and forwards

3. **Gateway**
   - Decapsulates the packet.
   - Looks up the flow in Routing Policy:
     - Match: `10.10.20.15:22/TCP`, action=FORWARD, publishers=[Publisher‑A, Publisher‑B].
   - Confirms user/device is allowed for this app.
   - Chooses the appropriate **Session Relay** in the Cloud Data Plane.

4. **Gateway → Session Relay**
   - Gateway forwards the SSH packet (and subsequent packets) to **Session Relay** over an internal secure channel.

Flow:  
**Client → Gateway → Session Relay**

#### Step 3: Session Relay picks a publisher

5. **Session Relay**
   - Has from Policy Orchestrator the mapping:  
     `service (10.10.20.15:22) → Publisher-A (primary), Publisher-B (secondary)`
   - Health checks / telemetry say **Publisher‑A** is healthy.
   - Session Relay forwards the traffic to **Publisher‑A**.

6. **Session Relay → Publisher‑A**
   - Uses its existing Session Relay–Publisher tunnel to send the SSH packets and associate them with this user/session.

Flow:  
**Client → Gateway → Session Relay → Publisher‑A**

#### Step 4: Publisher talks to the actual server

7. **Publisher‑A → App Server**
   - Publisher‑A is inside (or logically inside) the customer network, with reachability to `10.10.20.15`.
   - It initiates/handles a TCP connection from its local IP to:
     - `dst_ip = 10.10.20.15`
     - `dst_port = 22`
   - To the **App Server**, this looks like:  
     “A client at Publisher‑A’s IP is doing `ssh admin@10.10.20.15`.”

8. **TCP handshake completes**
   - SYN travels back and forth along:
     - **Client → Gateway → Session Relay → Publisher‑A → App Server**
     - And responses come back in reverse (detailed next section).

Once TCP is established, SSH starts its key exchange and interactive session over the same path.

Forward path summary:  
**Client → Gateway → Session Relay → Publisher‑A → 10.10.20.15 (SSH server)**

---

### 1.3. Data‑plane: app server → client (reverse)

Now the `10.10.20.15` server is sending SSH responses.

1. **App Server → Publisher‑A**
   - Server sees it has a TCP session with Publisher‑A’s IP.
   - Sends packets back to Publisher‑A as usual.

2. **Publisher‑A → Session Relay**
   - Publisher‑A sends those packets back inside its **Publisher–Session Relay** tunnel.
   - Tags packets with session metadata (service id, client session, etc.).

3. **Session Relay → Gateway**
   - Session Relay forwards them to the same **Gateway** that handled the original client tunnel.
   - No new policy decision — it’s just the reverse direction of an allowed session.

4. **Gateway → Client**
   - Gateway encapsulates packets back into the **client tunnel**.
   - Client decapsulates them and hands them to the OS TCP stack.
   - The `ssh` process on the laptop receives the data and shows it to the user.

Reverse path summary:  
**10.10.20.15 → Publisher‑A → Session Relay → Gateway → Client**

If Publisher‑A fails mid‑session:
- Session Relay/Gateway may fail the session; client may re‑establish via Publisher‑B, depending on design and capabilities (session stickiness vs reconnect) — but that’s an implementation detail on top of this same basic path.
